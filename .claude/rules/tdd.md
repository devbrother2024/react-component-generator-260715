# TDD Rule

> **이 규칙은 Rigid다 — 상황에 맞게 변형하지 마라.** "이번만 예외"는 없다. 예외가 필요하다고 느껴지면 아래 "변명 차단표"부터 확인한다.

## 0. 프로젝트별 규칙 우선

이 파일은 **기본값(fallback)** 이다. 프로젝트 루트의 `CLAUDE.md`/`AGENTS.md`에 TDD 관련 규칙(적용 범위, 예외, 도구, 커버리지 기준 등)이 명시돼 있으면 **그것을 우선** 따른다. 충돌하는 부분만 프로젝트 문서로 대체하고, 명시되지 않은 부분은 이 파일을 그대로 적용한다.

## 1. 적용 기준

**반드시 TDD 적용**

- 비즈니스 로직(도메인 규칙, 계산, 상태 전이)
- API 엔드포인트 / 서버 핸들러
- 유틸리티 함수(순수 함수, 데이터 변환)
- 버그 수정(재현 테스트 없이 수정 금지 — 회귀 방지가 핵심 목적)

**TDD 불필요**

- 타입 정의, 인터페이스 선언
- 설정 파일(빌드/린트/CI 설정 등)
- 순수 UI(마크업·스타일만 있고 로직 없는 컴포넌트)
- SQL/마이그레이션 스크립트 자체

대상이 애매하면 **"실패할 수 있는 로직이 있는가"** 로 판단한다. 있으면 적용 대상이다.

## 2. RED-GREEN-REFACTOR 사이클

### RED

- **하나의 동작 = 하나의 테스트.** 여러 동작을 한 테스트에 묶지 않는다.
- 테스트를 작성한 뒤 **반드시 실행해서 실패를 눈으로 확인**한다. 실행하지 않고 다음 단계로 넘어가지 않는다.
- 실패 이유가 **"기능 미구현"** 이어야 한다. 문법 오류, 타입 오류, import 오류로 실패한 것은 RED가 아니다 — 그런 실패는 먼저 고치고 나서 진짜 RED를 확인한다.

### GREEN

- **최소한의 코드만** 작성해 테스트를 통과시킨다. YAGNI — 지금 테스트가 요구하지 않는 기능은 만들지 않는다.
- **신규 테스트와 기존 테스트 전부** 통과하는지 확인한다. 신규 테스트만 통과시키고 넘어가지 않는다.

### REFACTOR

- 중복 제거, 이름 개선, 헬퍼 추출 등 구조 개선만 한다.
- 전 과정에서 **테스트는 계속 green 상태**를 유지해야 한다. 리팩터 도중 테스트가 깨지면 그 자리에서 고친다.
- **새로운 동작을 추가하지 않는다.** 새 동작이 필요하다고 느껴지면 리팩터를 끝내고 다음 RED로 넘어간다.

### 반복

- 다음 동작에 대한 **RED로 돌아간다.** 사이클을 건너뛰지 않는다.

## 3. 삭제 강제 규칙

- **테스트보다 프로덕션 코드를 먼저 작성했다면, 그 코드를 삭제하고 RED부터 다시 시작한다.** 이미 짠 코드가 아깝다는 이유로 순서를 건너뛰지 않는다.
- "일단 참고용으로 남겨두기"도 금지한다. 주석 처리, 다른 파일로 옮기기 등 어떤 형태로든 보존하지 않는다. 삭제하고 처음부터 다시 짠다.

## 4. 변명 차단표

| 변명 | 반론 |
|---|---|
| "너무 단순해서 테스트가 필요 없다" | **단순한 코드일수록 테스트 작성 비용이 낮다.** 비용이 낮은데 안 할 이유가 없다. 단순함은 생략의 근거가 아니라 실행의 근거다. |
| "나중에 테스트를 추가하겠다" | **나중은 오지 않는다.** 테스트 없는 코드는 이미 동작하는 것처럼 보이기 때문에 우선순위에서 항상 밀린다. |
| "시간이 없다" | **RED-GREEN 없이 짠 코드는 디버깅 시간으로 그 차이를 반드시 되갚는다.** TDD는 속도를 늦추는 게 아니라 나중에 낼 시간을 지금 내는 것이다. |
| "이미 짠 코드를 삭제하는 건 낭비다" | **검증되지 않은 코드는 자산이 아니라 부채다.** 순서를 지켜 다시 짜는 비용이, 검증 안 된 코드를 그대로 두는 리스크보다 항상 작다. |
| "이건 프로토타입이다" | **프로토타입은 반드시 버려지거나, 반드시 프로덕션이 된다.** 버려질 것이면 애초에 이 규칙의 적용 대상(1번 항목)이 아닌지 먼저 확인하고, 프로덕션이 될 것이면 지금부터 TDD를 적용한다. "일단 프로토타입으로"는 나중에 예외 없이 적용 대상이 된다. |
