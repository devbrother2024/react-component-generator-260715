# commit: 변경사항 분석 후 컨벤션 커밋

작업 디렉토리의 변경사항을 분석해 논리적 단위로 나누고, 각 단위에 맞는 한국어 커밋 메시지를 작성한 뒤 **사용자 승인을 받고** 커밋한다. 승인 없이 임의로 커밋하지 않는다.

## Step 0: 저장소·컨벤션 확인

- `git rev-parse --is-inside-work-tree`로 git 저장소인지 확인한다. 아니면 사용자에게 알리고 종료한다.
- 저장소 루트의 `AGENTS.md`(없으면 `CLAUDE.md`)에 커밋 메시지 컨벤션이 명시돼 있으면 그것을 우선 따른다. 없으면 이 스킬의 기본 컨벤션(아래 Step 2)을 사용한다.
- `git log --oneline -20`으로 최근 커밋 스타일을 확인해, 기존 프로젝트 관례(타입 접두사 유무, 언어, 어투)가 있으면 거기에 맞춘다.

## Step 1: 변경사항 파악

다음을 병렬로 확인한다:

- `git status --short` (untracked 포함, `-uall` 금지)
- `git diff` (unstaged 변경)
- `git diff --staged` (이미 staged된 변경)

확인할 것:

- **시크릿 의심 파일**: `.env`, `credentials.json`, `*.pem`, `*secret*`, `*key*` 등 민감해 보이는 파일이 변경/staged 목록에 있으면 내용을 먼저 열어보고, 실제 비밀값이 들어있으면 사용자에게 경고하고 그 파일은 사용자가 명시적으로 포함을 지시하지 않는 한 커밋 대상에서 제외한다.
- 변경사항이 전혀 없으면(`git status --short` 결과 및 diff 모두 비어있음) 그 사실을 알리고 종료한다(빈 커밋 금지).

## Step 2: 논리적 단위로 분류

변경 파일/hunk를 관심사 기준으로 그룹핑한다. 하나의 커밋에는 하나의 목적만 담는다(예: 기능 추가와 무관한 포맷팅 수정을 같은 커밋에 섞지 않는다).

그룹마다 타입을 하나 정한다:

| 타입 | 의미 |
|---|---|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 동작 변화 없는 구조/가독성 개선 |
| `chore` | 빌드 설정, 의존성, 잡무성 변경 |
| `test` | 테스트 추가/수정 |
| `docs` | 문서 변경 |

변경이 명확히 한 그룹이면 단일 커밋으로, 서로 다른 관심사가 섞여 있으면 여러 커밋으로 분리할 것을 제안한다.

## Step 3: 커밋 메시지 작성

형식: `{type}: {한국어 요약}`

- 요약은 한 줄, 간결하게(가능하면 50자 내외, 최대 72자). "무엇을 바꿨는지"보다 **왜 바꿨는지**가 드러나게 쓴다.
- 변경 배경이 diff만으로 설명되지 않을 만큼 복잡하면, 빈 줄 하나 두고 본문에 1-3줄로 이유를 보충한다.
- 실제 커밋될 내용(해당 그룹의 diff)만 근거로 작성한다. 다른 그룹이나 아직 반영되지 않은 변경을 언급하지 않는다.

## Step 4: 사용자 승인

커밋을 실행하기 전, 제안한 커밋 계획을 아래 형식으로 보여주고 승인을 받는다:

```
커밋 계획

1. [feat] {요약}
   포함 파일: a.ts, b.ts

2. [fix] {요약}
   포함 파일: c.ts
```

선택지:

- **전체 승인**: 계획대로 순서대로 커밋
- **메시지/그룹 수정**: 사용자 피드백을 반영해 계획을 다시 제시
- **취소**: 아무것도 커밋하지 않고 종료

승인받지 못한 그룹은 커밋하지 않는다.

## Step 5: 커밋 실행

승인된 그룹마다 순서대로:

1. 해당 그룹의 파일만 `git add {파일...}`로 스테이지한다. `git add -A`, `git add .` 금지 — 항상 파일을 명시한다.
2. 커밋 메시지는 개행이 깨지지 않도록 heredoc으로 전달한다:
   ```bash
   git commit -m "$(cat <<'EOF'
   {type}: {요약}

   {본문(있으면)}
   EOF
   )"
   ```
3. 커밋 직후 `git status --short`로 반영을 확인한다.
4. pre-commit hook 실패 시 `--no-verify`로 우회하지 않는다. 원인을 파악해 고치고, 필요하면 파일을 다시 스테이지한 뒤 **새 커밋**으로 재시도한다(직전 실패 시도는 커밋되지 않았으므로 `--amend` 금지).

모든 그룹 처리가 끝나면 최종 커밋 목록(해시 + 메시지)을 요약해 보여준다.

## 안전 규칙

- `git add -A` / `git add .` 금지. 항상 그룹에 속한 파일만 개별 지정한다.
- `--no-verify`, `--no-gpg-sign`, `-c commit.gpgsign=false` 등 훅·서명 우회 금지.
- `git commit --amend`는 사용자가 명시적으로 요청한 경우에만 사용한다. 기본은 항상 새 커밋.
- 이 스킬은 **커밋까지만** 담당한다. `git push`는 범위 밖이며, 사용자가 별도로 명시적으로 요청할 때만 수행한다.
- 시크릿으로 의심되는 파일은 사용자 확인 없이 포함하지 않는다.
